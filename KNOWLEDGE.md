## 1. 模板解析核心原则

### 1.1 占位符识别优先级

| 优先级 | 类型 | 示例 | 处理方式 |
|--------|------|------|----------|
| P0 | 精确匹配 | `<申办方>` `<项目名称>` | 精确字符串替换，必填 |
| P1 | 模糊匹配 | `XXXXXX公司` `XX例` | 只替换知识字段，不明确的标红 |
| P2 | 自由文本 | `[请在此处填写...]` | AI 生成，标记"需人工确认" |

**经验教训：** `<姓名>` 替换后可能残留 `<姓` 或 `名>` ——必须用全文精确匹配，不能逐字替换。

### 1.2 Run（文本块）处理

docx 段落可能拆成多个 run，直接遍历 runs 替换会漏：

```python
# ✅ 正确做法：段落级别合并 + 替换
def safe_replace(paragraph, old, new):
    full = ''.join(r.text for r in paragraph.runs)
    if old in full:
        for run in paragraph.runs:
            if old in run.text:
                run.text = run.text.replace(old, new)
                return True
    return False
```

### 1.3 段落对齐陷阱

**标题用 JUSTIFY（两端对齐）会导致短标题水平铺开，非常难看。**

```
❌ 两端对齐：研  究  终  点
✅ 左对齐：   研究终点
```

**规则：** 所有章节标题设为 LEFT，仅正文保留 JUSTIFY。

## 2. 字段提取策略

### 2.1 26 字段体系

```
基础信息：项目名称 / 方案编号 / 版本号 / 申办方 / 主要研究者
研究设计：研究类型 / 研究阶段 / 盲法 / 随机化 / 对照类型 / 样本量 / 研究人群 / 研究中心数
治疗方案：试验药 / 对照药 / 治疗周期 / washout期
终点指标：主要终点 / 次要终点 / 探索性终点 / 安全性指标
时间线：  启动日期 / 结束日期 / 期中分析时间点 / 最终分析时间点
```

### 2.2 提取可信度标注

- 🟢 可信（关键词+上下文明显匹配）
- 🟡 需确认（模糊匹配/多个候选值）
- 🔴 未提取（必须人工填写）

## 3. 文档生成质量守则

### 3.1 填充率阶梯

| 等级 | 填充率 | 要求 | 场景 |
|------|--------|------|------|
| S | ≥95% | 无红标无空章节 | 正式交付 |
| A | 80-94% | ≤10 红标 | 内部审查 |
| B | 60-79% | 需人工补充 | 初稿 |
| C | <60% | 不可用 | 知识库不足 |

### 3.2 必须避免的遗留问题

**语义断裂：** `"评估期中分析的和安全性"` → 错误合并，应为 `"有效性和安全性"`
**缩写不一致：** 同一文档 `TEAE` / `teae` / `不良事件` 混用 → 必须统一
**章节跳号：** 所有编号连续（1.1 → 1.2 → 1.3），跳号=解析错误
**残留指引：** `"[请在此处填写...]"` → 必须替换或标记

### 3.3 格式一致性清单

- [ ] 标题对齐 LEFT（非 JUSTIFY）
- [ ] 无残留 `<` `>` 符号
- [ ] 表格无溢出文字
- [ ] 页眉页脚已更新
- [ ] 字体统一（中文宋体，英文 Times New Roman）
- [ ] 字号规范（标题 18pt，正文 12pt，表格 10.5pt）
- [ ] 行距统一（固定值 20pt）
- [ ] 编号连续

## 4. 知识库生长机制

### 4.1 提炼闭环

```
文档生成 → AI 分析新内容 → 与知识库对比去重
→ 整理为结构化条目（治疗方案/终点定义/安全监测）
→ 推送给用户确认 → 确认后入库
```

### 4.2 知识条目格式

```json
{
  "id": "KB-2024-001",
  "type": "safety_monitoring",
  "disease_area": "胰腺癌",
  "source": "RWS-2024-012",
  "content": "伊立替康脂质体安全监测模式...",
  "confidence": 0.94,
  "confirmed": true
}
```

### 4.3 质量指标

- 匹配率 ≥85% → 健康
- 匹配率 60-85% → 需补充历史方案
- 匹配率 < 60% → 提醒用户批量导入

## 5. 多文档一致性保障

同一方案同时生成 DMC/DMP/SAP 时，以下字段必须严格一致：

| 字段 | 必须一致 |
|------|----------|
| 方案编号 | ✅ |
| 研究类型 | ✅ |
| 主要终点 | ✅（定义一致） |
| 期中分析时间点 | ✅ |
| 申办方 | ✅ |
| 安全性监测 | ⚠ DMC 最详细，以此为基准 |

版本同步：方案 v2.0 → 所有文档同时升级，不允许混合版本。

## 6. 异常处理

| 失败模式 | 原因 | 处理 |
|----------|------|------|
| 替换后格式丢失 | 跨 run 替换 | 合并 run 后替换，保留首个 run 格式 |
| 合并单元格失败 | docx XML 结构 | 用 cell.text 遍历 |
| OCR 乱码 | 扫描件 PDF | 提示文字版 PDF |
| 指引未被识别 | 非标准占位符 | 增加正则 `[请*]` 模式 |

**降级策略：** 知识库匹配失败 → LLM 推理（标注"AI 生成"）；模板解析失败 → 纯文本 `<XX>` 替换；OCR 差 → 只取置信度 >0.8 字段。

## 7. Crogo vs med-doc-generator 关键差异

| 维度 | 旧系统 | Crogo |
|------|--------|-------|
| 提取 | 纯正则 26 字段 | 知识库 RAG + AI |
| 模板 | 手动放目录 | 上传即解析 |
| 知识积累 | 无 | 批量建库 + 提炼闭环 |
| 多文档 | 独立生成 | 自动一致 |
| 校验 | 无 | 逐字段确认 |

**核心迁移：** 将 med-doc-generator 已验证的 26 字段提取 + 占位符替换算法作为基线，叠加知识库 RAG 提升准确率。

## 8. Cursor 开发建议

### 优先实现顺序

1. **模板解析引擎** — 上传 .docx 解析章节/占位符
2. **占位符替换引擎** — 合并 run → 精确匹配 → 保留格式
3. **26 字段提取** — 先正则规则，后叠加 RAG
4. **生成质检** — 自动检查格式/红标/空段
5. **知识库查询** — RAG 检索辅助填充
6. **知识提炼闭环** — AI 分析新知识 → 推送确认

### 技术推荐

- 后端：FastAPI（已验证稳定）
- PDF 解析：pymupdf（支持书签导航）
- docx 操作：python-docx
- 向量库：chromadb（轻量本地部署）
- 前端：纯 HTML+JS（无框架）
